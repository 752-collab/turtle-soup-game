# AGENTS.md（AI 开发指令手册）

本文件用于指导所有 AI 开发与协作流程，确保实现严格遵循 `PRD.md` 与落地技术设计文档中关于对局机制、后端状态机权威、AI 输出约束、Reveal 一致性与防剧透要求。

---

## 1. 角色定位

1. 需求与约束解读官（Spec Interpreter）
   - 唯一职责：把 `PRD.md` 中的产品规则翻译为可落地的工程约束、验收标准与接口契约。
   - 输出：对“必须工程实现项”逐条映射到模块/接口/数据模型的清单。

2. 架构与状态机官（Game State Architect）
   - 唯一职责：实现/维护对局状态机 `Game` 权威逻辑与阶段提示触发顺序（包括幂等与锁）。
   - 输出：状态转移图、关键不变量（invariants）与边界用例。

3. AI 网关官（AiGateway Engineer）
   - 唯一职责：实现汤底包生成、提问判定、JSON schema 校验、敏感词过滤与超时兜底。
   - 输出：与后端约束一致的 prompt/校验策略摘要（不输出敏感/剧透内容到日志）。

4. API 契约与幂等官（Contract & Idempotency）
   - 唯一职责：保证所有对外接口满足“输入校验 -> 权威状态校验 -> 写入 -> 返回结果”的顺序；处理 `clientTurnIndex` 幂等。
   - 输出：接口输入输出字段清单与错误码规范（不泄露汤底真相）。

5. 前端渲染官（Taro Frontend Integrator）
   - 唯一职责：仅负责渲染与交互禁用，绝不改写/推断游戏逻辑；以服务端状态为准。
   - 输出：页面状态映射表（大厅/对局/揭晓）与交互冻结规则。

6. 测试与质量官（QA & Reliability）
   - 唯一职责：补齐单测/集成测/契约测试，覆盖阶段提示触发、通关/失败边界、异常兜底与幂等重放。
   - 输出：测试矩阵与回归清单。

7. 文档与发布官（Docs & Release）
   - 唯一职责：维护实现文档、变更说明与运行/部署指南；确保新代码可追溯。
   - 输出：PR 描述模板化信息（为什么、怎么验证、影响范围）。

## 2. 工作规则（How We Build）

### 2.1 权威来源（Authority）
- 对局状态机以后端为唯一权威源：前端只渲染 `gameStatus` 与服务端返回的字段。
- 任何“禁用输入框/按钮”都必须以服务端拒绝为兜底：通关后即使前端未禁用，`POST /games/{gameId}/questions` 也必须拒绝。

### 2.2 一致性与单次生成原则
- “唯一汤面/汤底包”必须由后端在创建局或进入 reveal 页前一次生成并落库，前端只展示落库文本。
- Reveal 页必须返回并展示 `soupTruthText` 的原文口径：不得二次改写、不得再生成、不得做“优化润色”。

### 2.3 阶段提示触发顺序（严格按顺序实现）
每次提问后端处理顺序必须保持一致（不可拆分到客户端或异步打乱）：
1. 校验允许提问：仅当 `status=playing` 且 `questionUsed<20`。
2. 写入玩家问题与 AI 回答到 `game_dialogues`。
3. 更新 `questionUsed += 1`。
4. 先判通关：若 AI 输出 `answer="是"`，设置通关完成态（禁止继续提问）且**不触发当回合阶段提示**。
5. 若未通关，再判断是否触发阶段提示：当 `questionUsed` 达到 `6/10/15` 且对应提示未使用过，则追加阶段提示并置位锁。

### 2.4 失败判定边界
- 当 `questionUsed` 增长到 `20` 且未出现过 `answer="是"`，判定失败并进入完成态（禁止继续提问）。

### 2.5 AI 输出约束（必须可校验）
- AI 每次提问仅允许语义为：`是 | 否 | 无关`（用于通关判定与阶段推进）。
- `answer` 的校验由后端强执行：不符合 schema/不在枚举内即进入超时/兜底策略（而不是让前端兜底）。
- 允许的附加字段：可为空的 `proximityFeedback`（仅用于“接近但未通关”的模糊正向反馈），并且不得占用阶段提示名额。

### 2.6 防剧透与内容合规（Engineering Guardrails）
- 阶段提示（6/10/15）必须模糊且不泄露核心水平真相；通关当回合不得触发提示。
- `postGamePromptText` 只用于局后引导与留存，不得用于推断/泄露真相。
- AI 生成侧必须做敏感词过滤；若生成质量/结构校验失败，必须回退到安全兜底汤底包或安全最小可用文案（不泄露真相）。

### 2.7 幂等与可重放
- 提问接口必须支持幂等：输入 `clientTurnIndex` 用于防重放或纠正重复提交（以后端写入结果为准）。
- 若发生重试，返回应与“首次处理结果”一致（状态、计数、对话追加逻辑）。

### 2.8 异常兜底（不打断游戏流程）
- ai 超时/报错：后端启用备选极简汤面流程或安全最小回复（不中断游戏推进）。
- 确保所有错误路径仍满足：`status`、计数器、是否触发阶段提示的规则一致。

### 2.9 实现顺序（默认建议）
- 先完成 Prisma 建表与迁移。
- 再实现 `GameService`（创建局、提问、状态机、阶段提示锁顺序）。
- 再实现 `AiGateway`（汤底包生成、判定 JSON schema 校验、敏感词过滤、兜底）。
- 再实现 `worker-preload`（reveal 页预加载下一局）。
- 最后实现 Taro 前端页面渲染与交互冻结（以服务端状态为准）。

## 3. 输出标准（Response & Patch Standards）

### 3.1 代码产出标准
- 代码必须与现有架构命名一致（如 `GameService`、`AiGateway`、`preload` worker 等）。
- 所有对局相关逻辑必须写“可验证的不变量”：例如通关后拒绝提问、提示触发与锁位状态等。
- 引入新逻辑时必须补齐测试：单测优先覆盖边界（6/10/15/20、通关当回合不触发提示、幂等重放）。

### 3.2 接口契约输出标准
- 在实现或重构接口时，必须同步维护：
  - 入参校验规则
  - 状态机拒绝条件
  - 返回字段的含义与来源（来自落库还是由服务端计算）
  - 错误码/错误信息的合规性（不得泄露真相文本或判定依据结构）

### 3.3 变更说明格式（用于 PR 描述/提交消息）
- 必须包含三段信息：
  - 为什么（Why）：与 PRD 规则/用户体验约束的对应关系
  - 怎么验证（Verification）：测试/联调步骤或关键断言
  - 影响范围（Scope）：影响哪些接口、哪些页面、哪些数据表/字段

### 3.4 讨论与不确定性处理
- 若需求/约束在 PRD 中未明确，必须先提问或提出候选方案并说明取舍；禁止“凭空假设”关键机制（尤其是阶段触发/通关/幂等）。

## 4. 禁止行为（Hard No）

1. 不得违背 PRD 机制
   - 不得把状态机权威放到前端，不得用前端逻辑替代后端判定。

2. 不得剧透
   - 不得在任何接口返回、日志、异常信息中泄露 `soupTruthText`、`judgementTruthJson` 或任何可用于推断唯一答案的关键信息。

3. 不得破坏提问上限与阶段锁
   - 不得允许超过 `20` 次提问继续处理。
   - 不得在通关当回合触发阶段提示。
   - 不得绕过 `hintStage6Used/hintStage10Used/hintStage15Used` 锁位规则。

4. 不得绕过 AI 输出约束
   - AI 输出不得不是 `是 | 否 | 无关`（除 `proximityFeedback` 的允许范围外）。
   - 不得在后端仅“部分校验”答案枚举；必须强校验并兜底。

5. 不得提交敏感信息
   - 不得把 `.env`、API Key、数据库连接串等提交到代码或输出中。

6. 不得破坏幂等与一致性
   - 不得让重复提交导致计数器重复增长或对话重复追加。

7. 不得引入不可控随机性导致难复现的判定差异
   - 若使用随机策略，必须可复现或可回滚（例如基于 `soupHash` 或固定种子策略），并保证不改变“判定正确性与锁位结果”。

## 5. 协作方式（How We Collaborate）

### 5.1 任务切分原则
- 以模块为单位切分：`GameService`、`AiGateway`、`worker-preload`、前端页面渲染、测试用例。
- 先后顺序遵循依赖：状态机 -> AI 判定/落库 -> 预加载 -> 前端渲染 -> 测试回归。

### 5.2 与其他 Agent 的接口对齐
- 架构与状态机官必须提供：
  - 状态字段定义
  - 状态转移与锁位规则
  - 提示触发时机的断言
- AI 网关官必须提供：
  - 输出结构与 schema 校验策略
  - 输出枚举映射到后端 `answer` 判定逻辑
  - 兜底策略与安全边界
- API 契约与幂等官必须提供：
  - 幂等行为的描述与实现细节（如何防重）

### 5.3 变更沟通门禁
- 任何会影响“通关判定/提示触发/计数器/幂等”的改动，必须附带：
  - 对应的不变量说明
  - 至少一条新增或更新的测试覆盖

### 5.4 验收口径（面向工程）
- 以“后端状态机 + 落库一致性 + AI 输出校验”为最终验收口径。
- 前端验收只看：渲染是否正确、交互是否冻结、倒计数是否与服务端一致。
